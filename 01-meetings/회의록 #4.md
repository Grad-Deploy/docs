## II. 주요 결정 사항

### 1. 추가 기능 방향 확정

* 팀원 전체 논의를 통해 K8s 중심으로 기능 확장 방향을 합의함
* 네트워크 정책 설정, Ingress/Service 설정, DNS/서비스 디스커버리, 보안 기본값 관리 포함
* 이온이 K8s 관련 조사 자료(Grad-deploy-k8s.pdf)를 정리하여 팀 전체에 공유함
* 이은성이 논의 내용을 반영한 요구사항 명세서(v1.1) 및 과제 계획서(v2)를 작성하여 공유함
* 이혁제가 기존 프로젝트를 대상으로 직접 테스트를 진행하여 오류 메시지 관련 개선 필요성을 확인하고 팀에 공유함

### 2. CI/CD 도구 전환 결정

* 팀원 전체 검토 및 논의를 거쳐 Jenkins → GitHub Actions + Argo CD 방향으로 전환 합의함
* 기존 v1에서 GitHub Actions를 이미 사용 중이므로 Argo CD 추가 시 구조가 자연스럽게 연결됨
* Argo CD를 통해 비활성 시간대에 replicas를 0으로 설정하여 자원 낭비를 절감할 수 있음

### 3. K8s 운영 안정성 기능 추가

* 팀원 의견을 종합하여 아래 기능을 추가하기로 함
  * CPU Request/Limit 설정 자동 적용
  * Startup Probe 자동 적용, Terminating 파드 정리, HPA 안정화
  * Claude API를 활용한 K8s 오류 메시지 자연어 설명 기능 (이혁제 제안)

### 4. 향후 개발 일정

* 중간 보고서 작성 완료 (이온 초안 작성, 이은성 검토 및 확정)
* 중간 보고서를 기반으로 코드 모듈을 분리하고, 기능별 역할 분담 후 팀원 전체가 개발에 착수할 예정
* 역할 분담 확정 후 고민서를 포함한 전 팀원이 담당 모듈 개발을 병렬로 진행할 예정

## IV. 회의 내용

### 1. 추가 기능 아이디어 공유

* 이은성이 이전 피드백에서 기능이 다소 가볍다는 의견이 있었음을 공유하며, 팀원 전체에 추가 기능 아이디어를 요청함
* 이온은 실무에서 자주 발생하는 K8s 운영 문제를 해결하는 방향을 제안함; 자원 관리, 네트워크 정책 설정, `latest` 태그 사용 방지, 트래픽 관리 등을 언급하며, 포트/IP 자동 동적 할당 기능과 호스트 사양 기반 컨테이너 할당량 자동 제한 기능을 구체적인 예시로 제시함
* 이은성은 자체 테스트 과정에서 발견한 문제와 해결 방안을 팀에 공유함
  * SSH key 기반 GitHub Push 후 Actions 실행 시 SSH 서버 미설정으로 인한 오류 → Docker Hub 연동 방식으로 우선 해결
  * 데모데이 모드 URL 접속 불가 문제 → 백엔드 서버 API 연동 후 정상 동작 확인
* 이혁제는 기존 프로젝트를 직접 테스트하는 과정에서 K8s 배포 시 오류 메시지가 직관적이지 않은 점을 확인하고, Claude API를 활용하여 오류 메시지를 자연어로 설명해주는 기능 추가를 제안함
* 고민서는 일정을 공유하며 논의에 참여함

### 2. Jenkins → GitHub Actions + Argo CD 전환 논의

* 이은성이 K8s 집중 방향으로 Terraform + Jenkins 조합 활용을 제안하고, 논의 내용을 반영한 요구사항 명세서(v1.1)와 과제 계획서(v2)를 팀에 공유함
* 이온이 Jenkins 도입 시 우려되는 점을 조사하여 팀에 공유함
  * 대안으로 GitHub Actions + Argo CD 조합을 제안함
* 이은성은 Argo CD 도입 시 비활성 시간대 replicas를 0으로 설정해 자원 낭비를 줄일 수 있다는 점을 추가로 검토하여 공유하고 전환에 동의함
* 팀원 전체 검토를 거쳐 GitHub Actions + Argo CD 방향으로 전환하기로 최종 합의함

### 3. K8s 운영 안정성 기능 논의

* 이온이 실제 운영 환경에서 자주 발생하는 문제를 예방할 수 있는 안정성 기능 추가를 제안하고, 조사 자료(Grad-deploy-k8s.pdf)를 팀에 공유함
* 이은성은 파드 생성 중 오류 발생 시 재생성이 필요한 문제와 리소스 미설정으로 인한 운영 장애 등을 언급하며 해당 방향에 동의함
* 이혁제는 직접 테스트를 통해 경험한 오류 상황을 바탕으로 Claude API를 활용한 오류 메시지 자연어 설명 기능을 제안하였고, 팀 전체가 긍정적으로 검토함

### 4. 향후 개발 일정 논의

* 팀원 전체가 시험 종료 일정을 공유하며 개발 착수 일정을 함께 조율함
  * 이혁제: 4.20 종료 / 고민서: 4.24 종료 / 이은성: 4.22 종료 / 이온: 4.20 종료
* 이온이 중간 보고서(GradDeploy_중간보고서_v4.pdf) 초안을 작성하여 팀원들에게 공유하고 검토를 요청함
* 이은성이 검토 후 크게 수정할 부분이 없음을 확인하고, 현재 진행 상황 기록으로서 추후 변경이 있어도 무방하다는 의견을 제시함. 팀원 전원이 이에 동의함.
* 중간 보고서를 기반으로 코드 모듈을 분리하고, 기능별 역할 분담을 통해 팀원 전체가 담당 모듈 개발에 본격 착수하기로 함
* 개별 보고서 관련 사항은 추후 별도 공지 예정임을 안내함
